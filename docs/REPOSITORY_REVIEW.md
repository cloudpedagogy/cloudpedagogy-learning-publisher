# Repository Review: CloudPedagogy Learning Publisher

This review documents the state of the repository as inferred from its source code, CLI commands, templates, and tools as of the current implementation.

## What is this repository?
It is a Python-based publishing system (installed as `quarto-course-generator`) designed to generate Quarto-based educational sites and resources. It bridges structured configurations and Word documents with the Quarto publishing system.

## What problem does it solve?
Creating complex, multi-page interactive eLearning courses in Quarto requires significant boilerplate, precise file structuring, and complex Markdown/HTML authoring. This repository automates the scaffolding of Quarto sites and provides a specialized parser to convert specific authoring patterns in Microsoft Word documents into interactive web components (quizzes, callouts, R code blocks, etc.).

## Who is it for?
Instructional designers, subject matter experts, and educators who want to produce high-quality, interactive Quarto-based eLearning sites, but prefer authoring content in Microsoft Word and managing course structure via simple YAML configurations.

## What workflows are actually supported?
The codebase actually supports a **hybrid workflow** where both YAML and Word play distinct, interlocking roles:

1. **The Structural Workflow (YAML)**: Authors define the course hierarchy (modules, sessions, sections, pages) in a YAML configuration file. The `coursegen build` command uses Jinja2 templates to generate a complete directory structure of `.qmd` files, injecting navigation, metadata, and scaffolding.
2. **The Content Workflow (Word)**: Authors write the actual learning content in `.docx` files using a specific syntax (e.g., `R Code`, `Quiz`, `SelfCheck`). The `coursegen import_word` command converts these documents via Pandoc, parses the syntax into Quarto directives and HTML/JS interactions, and safely injects the content into the existing `.qmd` scaffold generated in step 1.
3. **The Direct Authoring Workflow**: Authors can skip the Word import and directly write Markdown/Quarto content into the generated `.qmd` scaffold.

## What outputs does it generate?
1. **Quarto Source Files**: `.qmd` files, `_quarto.yml`, and `styles.css`.
2. **Rendered HTML Website**: Generated by invoking Quarto via `coursegen render`.
3. **SCORM 1.2 Package**: A standalone script (`tools/package_scorm.py`) wraps the rendered HTML in a SCORM manifest and injects completion tracking JavaScript.
4. **PDF/Docx Handbooks**: A standalone script (`tools/build_handbook_from_quarto.py`) strips interactive web elements (like tabs and forms) from the `.qmd` files and compiles them into a single PDF or Word document.

## What functionality appears central?
- **YAML Configuration Parsing & Validation**: `ConfigLoader` and Pydantic models in `src/course_generator/models/`.
- **Course Scaffolding**: The `Generator` class in `src/course_generator/core/` and the Jinja2 templates in `templates/` are critical. Without the scaffold, there is no place to import Word content.
- **Word Import & Parsing**: The `import_word.py` tool is highly developed, with extensive regex parsing for custom syntax (e.g., `parse_quiz`, `parse_r_code`, `parse_tabs`). It represents the core content authoring experience.

## What functionality appears secondary?
- **Standalone Tools**: The SCORM packager and Handbook generator in `src/course_generator/tools/` act as post-processing scripts rather than integrated CLI commands.

## What functionality appears legacy?
- **Template Interactions**: The `templates/interactions/` directory contains Jinja templates for interactive components. However, `import_word.py` generates its own HTML/Markdown for these interactions directly via Python functions (e.g., `parse_quiz` generates `<form>` elements directly rather than invoking a Jinja template). This suggests the Jinja-based interaction templates might be a legacy approach from before the Word importer was fully built out.

## What functionality appears experimental?
- **WebR Integration**: The `parse_r_code` function in the Word importer supports `R Mode :: webr` to generate interactive WebR blocks. This appears to be a specialized, emerging feature.

## What functionality should be documented more prominently?
1. **The Interlocking Nature of YAML and Word**: Users must understand that they need to `build` the structure from YAML *before* they can `import_word`.
2. **The Word Authoring Syntax**: The specific text patterns required in Word (`R Code`, `Quiz`, `Option ::`, `Callout ::`) are heavily hardcoded in `import_word.py` and need comprehensive documentation for end-users.
3. **The SCORM and Handbook Tools**: These are highly valuable for enterprise/LMS deployments but are currently hidden as standalone Python scripts rather than integrated CLI commands.
